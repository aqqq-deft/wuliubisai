## 3. 已确认的硬件和软件环境

### 3.1 主板和底盘

- 主板：`CODBOT RCU v1.21`。
- MCU：`STM32F103ZE`，Cortex-M3。
- 底盘：四驱麦克纳姆轮，必须使用M100工程，不能套普通4WD差速算法。
- USB串口：`USB-SERIAL CH340`，115200、8N1；COM号以设备管理器当前显示为准，不能写死COM6。
- `S4`：BOOT0按键；右下角 `S2` 是普通KEY，不是BOOT。
- 右下角左侧小按钮：RESET。
- `LED+5V`：只表示存在5V供电。
- `LED/C14`：程序状态灯，对应PC14。

电池共同确认信息：标称10.8V、充电限制12.6V、最大持续放电6A。旧记录对容量写成过2550mAh和4550mAh，存在冲突，容量必须重新看标签确认。

四轮当前代码参数相同：`P=0.1、I=0.2、D=0.1`。

### 5.2 当前配置

```c
#define BUCKET_ENABLE_MOTION          0U
#define BUCKET_SERVO_NUMBER           1U
#define BUCKET_MIN_PULSE_US           1000U
#define BUCKET_MAX_PULSE_US           2000U
#define BUCKET_STEP_US                10U
#define BUCKET_DIG_PULSE_US           1500U
#define BUCKET_TRANSPORT_PULSE_US     1500U
#define BUCKET_DUMP_PULSE_US          1500U
```

含义：

- 机构真实动作暂时关闭。
- 三个位置全是1500μs占位值，尚未机械标定。
- 1000～2000μs只是代码初始保护范围，不代表机械结构能安全走满。
- 动作启用后，`Bucket_Update()` 每两个10ms控制周期改变10μs，即约每20ms走10μs（约500μs/s）。

### 5.3 重要：动作关闭不等于没有PWM

`Bucket_Init()` 即使在 `BUCKET_ENABLE_MOTION=0` 时，仍会调用：

```c
servoSetPluse(BUCKET_SERVO_NUMBER, 1500U);
```

所以舵机一旦正确接线并获得独立电源，主板启动后可能立即转向1500μs中位。首次必须拆摇臂或脱开连杆，让输出轴自由转动。不能把“动作宏为0”理解成“舵机绝不会动”。

### 5.4 启用动作前必须解决的通道冲突

当前 `auto_ControlCarStop()` 仍调用遗留的：

```c
auto_epsSteerControl(0, 0);
```

而 `steercontrol.c` 也会写第1路舵机。底盘停车、松开按钮、通信超时等情况可能反复把舵机1写回1500μs。现在三个机构位置恰好全是1500μs，所以问题未暴露；一旦标定成不同值，铲斗状态机就会与旧转向逻辑争抢同一PWM通道。

处理原则：

- 1500μs脱开摇臂的中位测试可以先做，因为两者目标一致。
- 在开始任何非1500μs标定之前，必须小改底盘停车逻辑，让M100底盘停车不再写舵机1；或者明确改用不冲突的舵机通道。
- 修改后重新编译、烧录，并回归验证底盘松手停车、500ms超时和舵机保持。
- 不要删除真正的四轮停车和PID清零，只移除与M100无关的旧转向舵机写入。

## 6. 舵机串口协议和状态机

控制链路：

```text
ROS2 /bucket_command
  → 串口命令码0x07
  → autocontrol.c
  → Bucket_HandleCommand()
  → bucket_control.c
  → servoSetPluse()
  → 舵机接口1 / PC6 / TIM8_CH1
```

命令帧现有格式：

```text
CD 02 07 cmd (07 XOR cmd)
```

状态反馈码为0x08，反馈帧格式：

```text
FD 04 08 state pulse_hi pulse_lo xor
```

命令值：

| 值 | 含义 |
|---:|---|
| 0 | 停止移动并保持当前位置 |
| 1 | 前往挖豆位 |
| 2 | 返回运输位 |
| 3 | 前往卸豆位 |
| 4 | 返回运输位 |
| 5 | 解除机构急停锁存并返回运输位 |
| 255 | 机构急停并锁存，之后必须发送5解除 |

状态值：

| 值 | 含义 |
|---:|---|
| 0 | 已停止 |
| 1 | 正前往挖豆位 |
| 2 | 已到挖豆位 |
| 3 | 正前往运输位 |
| 4 | 已到运输位 |
| 5 | 正前往卸豆位 |
| 6 | 已到卸豆位 |
| 255 | 机构急停锁定 |

底盘工具的“急停 / STOP”只发送底盘零速度帧，不等于舵机命令255。后续舵机工具必须分别提供“停止保持”“机构急停”和“复位解锁”。

特别注意：软件停止和机构急停只是把目标锁在当前脉宽，舵机仍输出PWM并保持扭矩，并不会切断舵机电源。机械卡死、持续啸叫、异常发热时必须物理切断舵机独立电源。

500ms没有有效串口控制帧时，当前主任务会停车并让铲斗进入急停锁存。将来制作标定工具时必须维持有效心跳，否则一次较长的平滑动作可能在0.5秒后中途停止，并需要命令5解除。

## 8. 下一步详细计划

### A. 断电确认接线

先画出明确接线表：主板接口1的信号、正极、GND；舵机线定义；独立5V级电源正负极；主控与舵机电源共地位置；确认不会把高电压送入Type-C 5V或电脑USB。

严禁把10.8V/12.6V主电池直接接到MG996R或标有5V的舵机电源轨。

### B. 脱开摇臂做1500μs中位测试

前提是接线照片已确认：

1. 小车架空，关闭主电源、Type-C和舵机独立电源。
2. 拆下摇臂或断开连杆，让输出轴自由。
3. 按确认后的接线表连接信号、独立舵机电源和共地。
4. 使用当前 `project2.hex`；暂不启用机构动作。
5. 按确认后的顺序给主控逻辑和舵机供电。
6. 舵机可能转到1500μs后保持。
7. 检查持续抖动、尖锐啸叫、异常发热、电源保护或掉压。
8. 任何异常立即切断舵机独立电源。

正常结果：只转一小段或保持原位，随后稳定保持，不持续来回抽动。

### C. 修复舵机1通道冲突

中位硬件测试通过后，先修改 `auto_ControlCarStop()` 对旧 `auto_epsSteerControl(0,0)` 的调用关系，确保M100底盘停车不再覆盖铲斗舵机1。保持四轮停车、PID清零、通信超时和其他安全逻辑不变。编译0错误、重新烧录，并再次验证底盘松手停车和1500μs保持。

### D. 制作安全标定工具/命令

当前底盘工具不能直接调舵机脉宽，现有机构协议也只有三个离散目标位且全为1500μs。因此需要实现一个最小范围的本地串口标定方式：

- 从1500μs开始，初始只允许小范围移动。
- 每次增加/减少10、20或50μs。
- 显示当前脉宽并严格做整数和范围校验。
- 有停止保持、机构急停、复位和持续心跳。
- 不发送底盘运动命令。
- 异常或断开时不能输出越界脉宽。

不要直接测试1000和2000μs两个极限。

### E. 脱开连杆标定舵机安全范围

从1500μs小步改变，先确认两侧方向；不要故意顶到内部机械止点。记录安全最小、最大值，并各留余量。断电后才安装摇臂。

### F. 连接机构标定三个位置

1. 将摇臂安装在接近运输位的位置。
2. 连接连杆后仍从1500μs附近小步调整。
3. 分别找出挖豆位、运输位、卸豆位。
4. 每个位置确认无顶死、过中心、卡住和拉扯线束。
5. 记录脉宽和照片。机械安全范围通常比舵机空载范围窄，以机构为准。

### G. 写入正式配置并启用

三个位置确认后，才填写真实 `MIN/MAX/DIG/TRANSPORT/DUMP` 脉宽，最后把 `BUCKET_ENABLE_MOTION` 从0改为1。

## 9. FlyMcu烧录要点

1. 选择设备管理器当前的 `USB-SERIAL CH340`，COM号可能变化。
2. 波特率115200，选择重新Build后的 `project2.hex`。
3. 勾选“校验”“编程后执行”“编程前重装文件”。
4. 不勾选“使用RamIsp”“连续烧录模式”“编程到FLASH时写选项字节”。
5. 断开Type-C，按住主板中央 `S4/BOOT`，插入Type-C后继续按约8秒再松开。
6. `S2`不是BOOT。
7. 烧录成功后关闭FlyMcu，避免占用CH340，再打开调试工具。

FlyMcu连续收到大量 `FD` 通常表示主板仍在运行应用程序并反馈，没有进入BootLoader，不代表芯片损坏。按RESET时CH340的COM口通常不会从Windows消失，这是正常的。

## 10. 已踩过的坑，绝对不要再踩

### 10.1 只靠电脑USB判断电机性能

C轮曾慢启动、略慢，根因是主电源没开，只靠Type-C/电脑供电。正常主电源开启后恢复。判断电机、编码器和PID前先确认电池、电源开关和电压。

### 10.2 把舵机接到主电池

主电池为10.8V、满充限制12.6V，不能直接送给设计使用独立5V级供电的舵机和舵机5V轨。必须确认极性、使用独立大电流电源并与主控共地。

### 10.3 以为 `BUCKET_ENABLE_MOTION=0` 就绝不会动

`Bucket_Init()`仍输出1500μs。舵机上电可能立即归中，首次必须拆摇臂或脱开连杆。

### 10.4 一开始就测试1000～2000μs全范围

代码范围不是机械安全范围。打到极限可能顶死、堵转、发热并损坏齿轮、摇臂或电源。必须从1500μs小步逼近。

### 10.5 忽略舵机1被两套代码同时控制

铲斗状态机和旧 `auto_epsSteerControl()` 都会写舵机1。当前全为1500才没暴露。非中位标定前必须消除冲突，否则底盘停车或通信超时会把铲斗突然拉回中位。

### 10.6 把软件急停当成断电

舵机STOP/急停只保持当前脉宽和扭矩，不切断电源。机构卡死、啸叫、发热时必须物理断开舵机电源。

### 10.7 PowerShell负数被当成字符串重复

旧工具参数没有强类型，负数曾变成大量 `-1-1-1...` 字符串，导致后退、横移、转向报 `Int32` 转换错误。现已使用强类型、显式Int32和命名参数，不要回退。

### 10.8 把左右横移映射换回旧值

实车验证后的正确工具映射是左横移Y=-1、右横移Y=+1。以实车为准，不要只凭直觉改回去。

### 10.9 在USART1中断或发送路径加入阻塞逻辑

早期连接串口后LED停止，问题在USART1接收/反馈路径。当前 `bsp_uart.c` 已针对COM1发送和中断修复。不要在中断中加入等待、阻塞发送、打印或复杂逻辑，也不要轻易恢复旧通用FIFO路径。

### 10.10 把RX十六进制滚动当成报错

工具中的 `RX FD ...` 是STM32反馈。只有程序灯停止、反馈中断、工具报错或动作异常时才排查。

### 10.11 用S2进入BootLoader

S4才是BOOT0。按错后FlyMcu收到FD并超时，不代表芯片坏。

### 10.12 使用ARM Compiler 6或安装Legacy Pack掩盖设置

厂家旧HAL在Compiler 6下会出现大量 `__weak` 等错误。当前工程必须用Compiler 5、Operating system=None；不要随意污染全局Keil环境。

### 10.13 把头文件放进结构体

`competition_config.h` 和 `bucket_control.h` 曾被放进 `LinkStatus_t` 内，造成数百个连锁错误。头文件必须位于结构体定义外。

### 10.14 破坏厂家源码GBK编码

厂家C/H含GBK中文注释，曾按UTF-8写回导致乱码。修改时保持原编码；新增纯英文配置文件可用UTF-8。

### 10.15 烧错旧固件或写选项字节

当前正式输出名是 `project2.hex`，不要误选旧 `project.hex`。绝对不要勾选“编程到FLASH时写选项字节”。

### 10.16 FlyMcu和调试工具同时占COM

一个COM口同一时间只能被一个程序使用。烧录后关闭FlyMcu，再开底盘或舵机工具。

### 10.17 把不同层次的成功混为一谈

- `0 Error(s)`只表示编译链接成功。
- FlyMcu写入100%只表示写入成功。
- LED闪烁只表示相应程序路径运行。
- 舵机转动不代表角度、供电、急停和机械安全全部通过。
- 每阶段记录真实步骤和现象，不能根据代码存在就宣布实物通过。

### 10.18 无依据提前调PID

电源不足、接触不良、机械摩擦和载荷变化都像轮速问题。先排除这些因素，在正常负载下采集可重复数据；没有量化问题不改PID。
